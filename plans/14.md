# Plan: Issue 14 — Laser Weapon

## Summary

Add a laser weapon to the turret. Unlike the existing bullet shooter (which fires moving projectile entities), the laser has **instantaneous beam propagation**: on each shot, a ray is cast from the barrel in the barrel's forward direction, the first destroyable entity hit takes damage immediately, and a short-lived visual beam is rendered from barrel to the impact point. The laser fires slowly (2 000 ms cooldown vs 500 ms for bullets).

No physics (ammo.js) involvement is needed — damage is applied directly via THREE.js raycasting, bypassing the `collidestart` pipeline entirely.

---

## Files to Change

| File | Change |
|------|--------|
| `components/laser-shooter.js` | **New.** A-Frame component that fires the laser: slow cooldown tick, THREE.js raycast for instant hit detection, direct damage call, beam entity spawn. |
| `components/destroyable.js` | Add a `takeDamage(amount)` public method so `laser-shooter` can damage an entity directly without going through collision events. |
| `index.html` | Add `<script>` tag for `laser-shooter.js`; add `laser-shooter` attribute to each turret cannon entity. |

`shooter.js`, `projectile.js`, `turret.js`, `control-switcher.js` — no changes needed.

## Implementation Steps

### 1. Add `takeDamage()` to `components/destroyable.js`

The existing `collidestart` handler calls internal damage logic inline. Extract it into a reusable public method:

```js
takeDamage: function(amount) {
    if (this.dead) return;
    this.data.health -= amount;
    if (this.data.health <= 0) this.die();
},
```

Update the existing `collidestart` handler to delegate to `takeDamage()`:

```js
// inside the collidestart handler
const targetEl = evt.detail.targetEl;
if (targetEl.classList.contains('bullet')) this.takeDamage(10);
```

This keeps the `bullet` path intact while opening a clean entry point for the laser.

### 2. Create `components/laser-shooter.js`

Register an A-Frame component `laser-shooter`:

**Schema:**
```js
schema: {
    cooldown:  { type: 'number', default: 2000 },  // ms between shots
    damage:    { type: 'number', default: 30 },
    range:     { type: 'number', default: 500 },    // units, matches projectile far-clip
    beamDuration: { type: 'number', default: 150 }, // ms beam stays visible
}
```

**State:** `this.lastShot = 0`, `this.raycaster = new THREE.Raycaster()`

**tick(time, timeDelta):**
1. If `time - this.lastShot < this.data.cooldown`, return.
2. Record `this.lastShot = time`.
3. Get world position (`p`) and world quaternion (`q`) from `this.el.object3D`.
4. Compute forward direction: `dir = new THREE.Vector3(0, 0, -1).applyQuaternion(q).normalize()`.
5. Set `this.raycaster.set(p, dir)`.
6. Collect all meshes in the scene (`this.el.sceneEl.object3D` traverse, skip the turret's own subtree to avoid self-hit).
7. Call `this.raycaster.intersectObjects(meshes, false)`. Take the closest intersection within `this.data.range`.
8. **If hit:**
   - Walk up the Three.js object hierarchy to find an A-Frame entity (`object.el`).
   - Check if that entity has a `destroyable` component.
   - If yes, call `entity.components.destroyable.takeDamage(this.data.damage)`.
   - Set `hitPoint` to `intersection.point`.
9. **If no hit:** Set `hitPoint` to `p + dir * this.data.range`.
10. Spawn a beam visual (see below).
11. Play sound if `this.el.components.sound` exists.

**`_spawnBeam(origin, hitPoint)` helper:**
- Compute `midpoint` = average of `origin` and `hitPoint`.
- Compute `length` = distance between them.
- Create `<a-entity>`:
  - `geometry="primitive: cylinder; radius: 0.02; height: {length}"`
  - `material="color: #F00; emissive: #F00; emissiveIntensity: 2; opacity: 0.85; transparent: true"`
  - `position` = midpoint
  - Rotate cylinder to align with `dir`: cylinders default to Y-up; apply a quaternion rotation so Y axis aligns with `dir`. Use `THREE.Quaternion.setFromUnitVectors(new THREE.Vector3(0,1,0), dir)` converted to Euler for the `rotation` attribute.
- Append to `this.el.sceneEl`.
- After `this.data.beamDuration` ms, remove the entity (`setTimeout`).

### 3. Register and add to `index.html`

Before closing `</head>`, add:
```html
<script src="components/laser-shooter.js"></script>
```

On each cannon entity (the same entities that already have `shooter`), add `laser-shooter`:
```html
<a-entity id="left-cannon" turret="" shooter="" laser-shooter="" ...>
```

Both turrets fire simultaneously (same behaviour as `shooter`).

## Testing

No automated tests exist; verification is manual on desktop (mouse mode) and Meta Quest:

1. **Fire rate**: Laser fires once every ~2 s. Holding the mouse button (or continuously playing) does not produce rapid-fire beams.
2. **Visual beam**: A red glowing cylinder appears briefly from each barrel tip to the first obstacle/enemy hit, then disappears after ~150 ms.
3. **Instantaneous hit**: Beam visually reaches the target on the same frame it fires — no travel delay observable.
4. **Hit detection**: An enemy directly in the turret's line of fire takes 30 damage. A 20 HP enemy dies in one shot; a 60 HP enemy requires three shots.
5. **Range limit**: Laser fired into empty space spawns a beam extending to `range` units; no crash or lingering entity.
6. **No self-hit**: The turret structure itself does not receive damage when the laser fires.
7. **Beam cleanup**: After `beamDuration`, the beam entity is removed from the DOM. Firing many times leaves no orphan entities.
8. **Bullet coexistence**: The existing rapid-fire bullet shooter continues to work alongside the laser; both can hit and destroy enemies.
9. **Destroyable backward compat**: Bullet-hit `collidestart` still deals 10 damage and kills enemies as before.
10. **VR**: On Meta Quest, laser fires from both turrets when in mouse mode (single cursor); in dual-controller mode, each barrel fires independently.

## Risks / Edge Cases

- **Mesh traversal cost**: Calling `raycaster.intersectObjects` on all scene meshes every shot is acceptable at the 2 000 ms cooldown (not per-frame). However, if the scene is large, scope the traversal to entities with the `destroyable` component by pre-querying `document.querySelectorAll('[destroyable]')` and walking their `object3D` children only.
- **Self-intersection**: The ray originates at the barrel's world position. If the barrel mesh is included in the intersection test, it will always register a hit at distance ≈ 0. Guard by either skipping the turret's own `object3D` subtree during traversal or setting `raycaster.near = 0.5`.
- **Cylinder orientation**: A-Frame cylinders are Y-up. Aligning them to an arbitrary 3D direction requires converting a Three.js quaternion to Euler angles for the `rotation` attribute — floating-point Euler gimbal lock is not an issue here because we set quaternion first. Alternatively, set `object3D.quaternion` directly after appending the entity.
- **`takeDamage` guard**: The `if (this.dead) return` guard in `takeDamage()` prevents double-kill if the laser and a bullet hit the same frame.
- **No physics needed**: Laser bypasses ammo.js entirely. This means the beam does not interact with static `ammo-body` walls (it only hits mesh geometry). This is fine for the current scene where walls/floors are not destroyable, but if collidable non-destroyable objects are added later, they will still occlude the laser (mesh intersection catches them; damage is simply not applied since they have no `destroyable` component).
- **Sound asset**: `assets/laserShoot.wav` already exists in the project and is referenced in `index.html`. The laser shooter can reuse it at a different playback position or just reuse the same asset.
- **Two cannons firing simultaneously**: Both left and right cannons have independent `laser-shooter` components with their own `lastShot` timers. They fire at the same interval but may drift slightly out of phase if the page is loaded mid-tick. This is acceptable cosmetically.
- **className check in destroyable.js**: The existing hit handler uses `classList.contains('bullet')` (or exact match). After refactoring to `takeDamage()`, ensure the `bullet` path still works correctly — the collision handler should call `takeDamage(10)` directly rather than duplicating the damage logic.
