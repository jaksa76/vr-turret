# Plan: Issue 13 — Damage Powerup

## Summary

Add a collectible damage powerup that floats in the play area. When a player bullet hits it, it activates a temporary damage multiplier (2×, lasting 10 seconds) on both turrets. While active, each bullet deals 20 damage instead of the default 10, killing most enemies in one shot. The powerup respawns at its original position after a cooldown (20 s). A subtle HUD indicator (colored overlay or text) signals the buff is active.

---

## Files to Change

| File | Change |
|------|--------|
| `components/powerup-damage.js` | **New.** A-Frame component that manages the powerup entity: rotating visual, bullet collision pickup, event emission, self-hide + respawn timer. |
| `components/shooter.js` | Track active damage multiplier; stamp each spawned bullet with a `data-damage` attribute. Listen for scene events to set/clear the multiplier. |
| `components/destroyable.js` | Read damage from `evt.detail.targetEl.dataset.damage` instead of hardcoding `10`. |
| `index.html` | Add `<script>` for `powerup-damage.js`; place one or more powerup entities in the scene. |

## Implementation Steps

### 1. Modify `components/destroyable.js` — read damage from bullet

In `hit()`, replace the hardcoded `10` with the bullet's `data-damage` attribute:

```js
hit: function(evt) {
    if (evt.detail.targetEl.className == 'bullet') {
        const dmg = parseInt(evt.detail.targetEl.dataset.damage) || 10;
        this.data.health -= dmg;
        if (this.data.health <= 0) {
            this.die();
        }
    }
    // ...
}
```

This is a pure backward-compatible change: bullets without `data-damage` still deal 10.

### 2. Modify `components/shooter.js` — stamp bullets with current damage

Add multiplier state in `init()` and listen for powerup events on the scene. When spawning a bullet, set `dataset.damage` on the element before appending.

```js
init: function() {
    this.lastShot = 0;
    this.damageMultiplier = 1;
    this._clearMultiplier = null;

    this.el.sceneEl.addEventListener('damage-powerup-activated', (evt) => {
        const { multiplier, duration } = evt.detail;
        this.damageMultiplier = multiplier;
        clearTimeout(this._clearMultiplier);
        this._clearMultiplier = setTimeout(() => {
            this.damageMultiplier = 1;
        }, duration);
    });
},
```

In `tick()`, after `projectile.setAttribute('class', 'bullet')`, add:

```js
projectile.dataset.damage = 10 * this.damageMultiplier;
```

### 3. Create `components/powerup-damage.js`

Register component `powerup-damage`:

**Schema:**
```js
schema: {
    multiplier:  { type: 'number', default: 2 },
    duration:    { type: 'number', default: 10000 },  // ms buff lasts
    respawnDelay:{ type: 'number', default: 20000 },  // ms until it reappears
}
```

**`init()`:**
- Build the visual as a child entity: an octahedron (or rotated box) in bright yellow/orange with `emissive` material, scale ~0.4.
- Attach physics for bullet pickup: `ammo-body="type: kinematic"` + `ammo-shape="type: sphere; radius: 0.4"` on the component's element.
- Add a continuous Y-rotation animation via A-Frame `animation` component on the visual child.
- Bind `collidestart` on `this.el` to `this.onHit`.

**`onHit(evt)`:**
- Check `evt.detail.targetEl.className == 'bullet'`.
- If yes, call `this.activate()`.

**`activate()`:**
- Emit `'damage-powerup-activated'` on `this.el.sceneEl` with `{ multiplier, duration }`.
- Hide the entity: `this.el.object3D.visible = false` and remove physics temporarily (set `ammo-body` type to `static` or toggle a flag to skip `onHit` while inactive).
- Schedule respawn: `setTimeout(() => this.respawn(), this.data.respawnDelay)`.

**`respawn()`:**
- Re-show entity: `this.el.object3D.visible = true`, restore `onHit` guard.

**Visual design note:** Use `geometry="primitive: octahedron; radius: 0.3"` with `material="color: #FFD700; emissive: #FF8800; emissiveIntensity: 1.5"`. Optionally add a `<a-text>` child showing "2×DMG" floating above it for clarity in VR.

### 4. Update `index.html`

Add script tag (before closing `</head>` or with other component scripts):
```html
<script src="components/powerup-damage.js"></script>
```

Place one or more powerup entities in the scene at positions visible and reachable by turret fire:
```html
<a-entity
    powerup-damage="multiplier: 2; duration: 10000; respawnDelay: 20000"
    ammo-body="type: kinematic"
    ammo-shape="type: sphere; radius: 0.4"
    position="0 1.5 -15">
</a-entity>
```

Consider adding a second powerup at a different position so both turrets have one accessible from their angle.

### 5. (Optional) Active buff indicator

Add a simple visual cue while the buff is active. In `shooter.js`'s event listener, toggle a CSS class or `data-attribute` on a HUD element, or change the turret material briefly. The minimal version: do nothing beyond the gameplay effect and trust player recognition.

## Testing

No automated test suite; verification is manual on desktop browser and Meta Quest:

1. **Default damage**: Without collecting the powerup, bullets still deal 10 damage. A 20 HP enemy dies in 2 shots, a 30 HP enemy in 3 shots — unchanged from before.
2. **Pickup by bullet**: Shoot the floating powerup; it disappears and the buff activates.
3. **Increased damage**: After pickup, a 20 HP enemy dies in 1 shot (20 damage per bullet). A 30 HP enemy dies in 2 shots.
4. **Buff expiry**: After 10 seconds, damage returns to 10. Enemies that required 1 shot now require 2 again.
5. **Respawn**: 20 seconds after being collected, the powerup reappears at its original position.
6. **Re-collect during active buff**: Shooting the respawned powerup resets the 10 s timer (no infinite stacking).
7. **Both turrets buffed**: Both left and right turrets deal boosted damage after a single pickup (both listen to the scene event independently).
8. **No self-hit / no crash**: Shooting the powerup does not destroy any enemy or produce an error. The bullet is consumed by the collision as normal.
9. **Existing destroyable backward compat**: All existing enemy kill paths (bullet collision) still work correctly after the `destroyable.js` change.

## Risks / Edge Cases

- **`data-damage` timing**: The bullet's `dataset.damage` must be set before ammo.js processes the physics tick that generates the collision event. Appending the element to the DOM and setting `dataset` on the same JS frame should be safe, but verify if the collision callback fires on a different tick.
- **Multiplier stacking**: If two powerups are placed and both are shot in rapid succession, the `clearTimeout` + reset pattern in `shooter.js` ensures only the most-recent buff duration counts (no permanent boost). Explicitly test this.
- **Both shooters share the scene event**: Since left-cannon and right-cannon each have their own `shooter` component, each independently subscribes to `'damage-powerup-activated'`. This is correct — both get buffed. But `removeEventListener` should be called in `remove()` to avoid leaking listeners if the entity is ever removed.
- **`ammo-body` on pickup**: The powerup entity needs a kinematic body and shape to receive `collidestart` from bullets. Ensure the shape radius is large enough to be hittable but not so large it overlaps enemy paths. Consider using `type: static` instead of `kinematic` — static bodies don't need per-frame updates and work for stationary collectibles.
- **Visual obscuring enemies**: Place powerups off to the side or above the main flight paths so they don't accidentally block sight lines.
- **`destroyable.js` class check**: The existing check uses `className == 'bullet'` (exact string equality, not `classList.contains`). The powerup's `onHit` should use the same check to stay consistent. Do not change the bullet class or this check will break.
- **Interaction with plan 14 (laser-shooter)**: Plan 14 adds a `takeDamage(amount)` method that is called directly (not via `collidestart`). The laser does not go through `dataset.damage` on the bullet. If the damage multiplier should also apply to the laser weapon, `laser-shooter.js` will need to read the multiplier from a shared source (e.g., `this.el.sceneEl.dataset.damageMultiplier`) rather than using a hardcoded value. This coordination is out of scope for this issue but should be noted.
