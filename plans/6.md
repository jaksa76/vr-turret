# Plan: Issue 6 — Shooting Enemy

## Summary

A `shooting-enemy` component and a `projectile-motion` component already exist in `components/shooting-enemy.js`, and `enemy-spawner.js` already has rudimentary support for spawning one shooting-enemy wave. However, three things are missing:

1. **Configurable 3D model** — the model is hardcoded to `#pancake-ship` inside enemy-spawner; the `shooting-enemy` component has no model schema property.
2. **Self-contained component** — enemy-spawner manually appends a model child with destroyable/physics attributes. This should be the component's responsibility so the model can be configured as a schema property.
3. **More waves** — only a single shooting enemy is spawned. Several distinct waves are needed.

The plan moves model creation (including `ammo-body`, `ammo-shape`, and `destroyable`) into `shooting-enemy.init()`, adds a `model` schema property, updates `enemy-spawner` to pass `wave.model` through, and expands the waves array.

---

## Files to Change

| File | Change |
|------|--------|
| `components/shooting-enemy.js` | Add `model`, `modelScale`, `health` schema props; create child model entity in `init`; remove the entity when the enemy is destroyed |
| `components/enemy-spawner.js` | Stop manually creating the model child for shooting-enemy; pass `wave.model`, `wave.modelScale`, `wave.health` through to the component; add new shooting-enemy waves |
| `index.html` | No structural changes needed; the existing `#pancake-ship` asset covers all waves. If a second model is added later, register it here as an `<a-asset-item>` |

## Implementation Steps

### 1. Add schema properties to `shooting-enemy`

In `components/shooting-enemy.js`, extend the schema:

```js
model:      { type: 'string', default: '#pancake-ship' }, // asset selector
modelScale: { type: 'vec3',   default: { x: 0.5, y: 0.5, z: 0.5 } },
health:     { type: 'number', default: 20 },
```

### 2. Move model/physics/destroyable creation into `shooting-enemy.init()`

At the end of `init`, create the child model entity the same way enemy-spawner currently does:

```js
const model = document.createElement('a-entity');
model.setAttribute('gltf-model', this.data.model);
model.setAttribute('scale', `${this.data.modelScale.x} ${this.data.modelScale.y} ${this.data.modelScale.z}`);
model.setAttribute('ammo-body', { type: 'kinematic', emitCollisionEvents: true });
model.setAttribute('ammo-shape', { type: 'box', fit: 'manual', halfExtents: '2 .5 2' });
model.setAttribute('destroyable', 'health', this.data.health);
this.el.appendChild(model);
```

This makes the component self-contained: a single entity declaration in HTML (or a single `setAttribute` call from enemy-spawner) is all that is needed.

### 3. Remove manual model creation from `enemy-spawner.js` (shooting-enemy branch)

Delete the code that currently creates the `model` child entity and appends `ammo-body`, `ammo-shape`, and `destroyable` in the `shooting-enemy` case. Pass the new props instead:

```js
enemy.setAttribute('shooting-enemy', {
    curve:        wave.curve,
    stopProgress: wave.stopProgress,
    fireRate:     wave.fireRate,
    target:       wave.target,
    model:        wave.model || '#pancake-ship',
    modelScale:   wave.modelScale || { x: 0.5, y: 0.5, z: 0.5 },
    health:       wave.health || 20,
});
```

### 4. Expand the `waves` array in `enemy-spawner.js`

Replace the single shooting-enemy wave with several waves that use different paths, fire rates, enemy counts, and (optionally) models:

```js
// Wave 2 — first shooting enemies appear from the left
{ start: 5000,  number: 2, interval: 2000, type: 'shooting-enemy',
  curve: '-10 -30 -80, -15 0 -50, -10 15 0, -10 0 20',
  stopProgress: 0.55, fireRate: 0.4, target: '#camera',
  model: '#pancake-ship', health: 30 },

// Wave 3 — faster-firing enemies from the right
{ start: 15000, number: 2, interval: 2500, type: 'shooting-enemy',
  curve: '10 -30 -80, 15 0 -50, 10 15 0, 10 0 20',
  stopProgress: 0.55, fireRate: 0.8, target: '#camera',
  model: '#pancake-ship', health: 30 },

// Wave 4 — mixed: fly-along-curve distraction + high-HP shooting enemy
{ start: 25000, number: 4, interval: 500,  type: 'fly-along-curve',
  curve: '5 -30 -80, 5 0 -50, 5 15 0, 5 0 30' },
{ start: 26000, number: 1, interval: 1000, type: 'shooting-enemy',
  curve: '0 -30 -80, 0 0 -50, 0 10 0',
  stopProgress: 0.6, fireRate: 1.0, target: '#camera',
  model: '#pancake-ship', health: 50 },
```

Tune timings and curves after playtesting.

### 5. (Optional) Add a second 3D model for visual variety

If a second `.glb` file becomes available (e.g. a fighter model), register it in `index.html`:

```html
<a-asset-item id="fighter-ship" src="assets/Fighter.glb"></a-asset-item>
```

Then reference it in wave configs: `model: '#fighter-ship'`. No code changes required beyond having the asset registered.

## Testing

This is a browser/WebXR app with no automated test suite, so all testing is manual.

| Scenario | Expected result |
|----------|----------------|
| Load `index.html`; wait 5 s | First shooting-enemy flies its curve and stops at `stopProgress` |
| Wait for enemy to stop | Enemy rotates to face `#camera` each frame |
| Enemy shoots | Red square projectiles (`projectile-motion`) travel toward camera at configured speed |
| Shoot the enemy with turret | Enemy takes damage (10 HP/bullet); dies and spawns explosion at 0 HP |
| Change `model` in wave config to a different asset | New model appears on the enemy entity |
| Change `health` in wave config | Enemy requires proportionally more/fewer shots to kill |
| Multiple shooting-enemy waves | Enemies spawn at correct relative start times; paths and fire rates match wave definitions |
| Shoot enemy before it stops | Destroyable health still works while enemy is in motion |
| Enemy projectiles reach lifetime | Projectiles despawn after `projectileLifetime` ms |

## Risks / Edge Cases

- **`ammo-body` timing** — The child model entity must be appended to the DOM before `ammo-body` initialises. Creating the child in `init` (before the physics tick runs) should be fine, but if the scene isn't ready yet the physics body might not register. Guard with an `'loaded'` event listener on `this.el.sceneEl` if this proves problematic.

- **`destroyable` on child vs parent** — Currently `destroyable` lives on the model child, and `collidestart` fires on that child when a bullet hits it. This is consistent with the existing `fly-along-curve` pattern and should work without changes to `destroyable.js`.

- **Enemy projectiles deal no damage** — `projectile-motion` has no collision detection; enemy shots are currently visual-only. This is a known limitation (noted in `shooting-enemy.js` comments). Out of scope for this issue but worth flagging.

- **Wave restart / no game loop** — Waves only fire once (via `setTimeout` in `init`). If a future "restart" feature is added, `enemy-spawner` will need to cancel and re-register its timers.

- **`stopProgress` near 0 or 1** — A value very close to 0 may cause the enemy to stop immediately before moving, looking odd. The existing clamp to `0.9999` prevents stopping exactly at the end. Recommend keeping `stopProgress` in the range `0.3–0.8` in wave definitions.

- **Model asset not preloaded** — If `wave.model` references an asset ID that hasn't been added to `<a-assets>`, the model will silently fail to render. Document the requirement that all model selectors must be registered as `<a-asset-item>` in `index.html`.
