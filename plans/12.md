# Plan: Faster Shooting Powerup [Issue #12]

## Summary

Add a collectible powerup that temporarily increases the turrets' fire rate. Enemies will drop it on death (with some probability). The player collects it by proximity (walking near it or shooting it). When active, both turrets fire at 2× speed for a fixed duration, with a visual indicator on the powerup entity while active.

---

## Files to Change

| File | Change |
|------|--------|
| `components/shooter.js` | Replace hardcoded `500`ms threshold with a data property; add buff/reset logic |
| `components/destroyable.js` | Spawn a powerup entity on death with some probability |
| `components/powerup.js` | **New file** — floating pickup, proximity collection, buff dispatcher |
| `index.html` | Add `<script>` tag for `powerup.js` |

## Implementation Steps

### 1. Refactor `shooter.js` to support variable fire rate

- Add a schema property `fireInterval` (default `500`, in ms).
- Replace the hardcoded `500` in the `tick` function: `if (time - this.lastShot > this.data.fireInterval)`.
- Add a method `applySpeedBuff(duration, multiplier)` that:
  - Calculates the boosted interval: `this.data.fireInterval / multiplier` (e.g. half = 2× speed).
  - Stores the original value.
  - Sets `this.data.fireInterval` to the boosted value.
  - Uses `setTimeout` to restore the original value after `duration` ms.
  - Guards against overlapping buffs (e.g. reset the timeout if a second powerup is collected before expiry).

### 2. Create `components/powerup.js`

- Schema: `duration` (default 8000 ms), `multiplier` (default 2), `dropChance` (not used here; just the pickup logic).
- `init`: Create a visually distinct child entity (e.g. a small glowing yellow sphere using `<a-sphere>`, color `#FFD700`, emissive glow via material `emissive`). Add a floating bob animation using A-Frame's `animation` component (position y oscillates ±0.15).
- `tick`: Each frame compute distance from `#camera`. If within ~0.6 units, trigger collection.
- Collection logic:
  - Find all entities with `[shooter]` via `document.querySelectorAll('[shooter]')`.
  - Call `el.components.shooter.applySpeedBuff(duration, multiplier)` on each.
  - Play a pickup sound (reuse `#laserShoot` or add a new `<audio>` asset).
  - Remove (or hide) the powerup entity.
- Visual feedback: while powerup is on the ground, rotate slowly (`animation` on rotation y).

### 3. Modify `destroyable.js` to drop powerups on death

- In the `die()` method, after removing the entity, roll a random chance (e.g. 40%): `Math.random() < 0.4`.
- If triggered, create a powerup entity at the dead enemy's world position:
  ```js
  const pu = document.createElement('a-entity');
  pu.setAttribute('powerup', '');
  pu.setAttribute('position', worldPos);
  sceneEl.appendChild(pu);
  ```
- Keep it simple — no extra attributes needed; defaults handle duration and multiplier.

### 4. Update `index.html`

- Add `<script src="components/powerup.js"></script>` alongside the other component script tags.
- Optionally place one powerup in the scene for easier testing during development (e.g. at `position="0 0.5 -3"`), then remove it once enemy drops work.

## Testing

Since there is no automated test suite, verification is manual:

1. **Fire rate baseline** — Open the game, count shots per second before any powerup (~2/s). Confirm `fireInterval` data property is respected.
2. **Powerup spawn** — Kill enough enemies to observe a powerup drop (or place one manually in the scene). Confirm the floating/rotating visual appears.
3. **Collection** — Walk the camera into the powerup sphere. Confirm it disappears and shot rate roughly doubles (~4/s).
4. **Duration expiry** — After 8 seconds confirm fire rate returns to normal.
5. **Double-collect** — Collect a second powerup before the first expires; confirm the timer resets cleanly (no double-speed-doubling or permanent buff).
6. **Both turrets** — Confirm both the left and right turret fire faster, not just one.
7. **VR mode** — Test on Meta Quest; proximity collection should work the same way since camera pose is tracked.

## Risks / Edge Cases

- **Double-buff stacking:** If the player collects two powerups before expiry, the naive `setTimeout` approach could restore the wrong original value. Guard against this by always saving the _base_ interval (not the current buffed one) and clearing any pending timeout before setting a new one.
- **Powerup persists after scene reload:** Entity is appended to the scene dynamically; a page reload clears it, so no persistent state issue.
- **Proximity collection in VR:** The camera rig moves via VR headset tracking, so distance-to-`#camera` works correctly. No special VR input needed.
- **Audio pool saturation:** Collection plays a sound at the same time bullets may be firing. Using a separate short audio asset (or the existing laser pool) should avoid audio glitches.
- **Enemy drops powerup at z < -500:** Projectile cleanup removes bullets at z < -500; enemies are destroyed much closer to the player, so powerup drop positions should always be in-bounds.
- **Multiple enemies dying simultaneously:** Each enemy independently rolls the drop chance, so multiple powerups could exist at once. Each powerup independently calls `applySpeedBuff`, which resets the timeout; the last pickup wins, which is fine.
