# Plan: Issue 10 — Game Over Screen

## Summary

Add a game over screen that appears when all enemy waves have been defeated. The overlay shows a "Game Over" message and a "Play Again" button that restarts the game via `location.reload()`.

The trigger is the completion of all scheduled waves. `enemy-spawner.js` already chains all spawns via `setTimeout`; the final timeout just needs to emit a `waves-complete` scene event after a short grace period. A new `game-state.js` component listens for that event and reveals a fixed-position HTML overlay.

This plan deliberately keeps scoring out of scope (that belongs to issue 11). The `game-state` component introduced here is designed to be extended by issue 11 with score accumulation.

| File | Change |
|------|--------|
| `components/enemy-spawner.js` | After the last wave's final spawn timeout fires, emit `waves-complete` on the scene (with ~2 s grace period) |

| `index.html` | Add `<script>` for `game-state.js`; attach `game-state` to the spawner entity; add the `#game-over-overlay` div to `<body>` |

## Implementation Steps

### 1. Emit `waves-complete` — `enemy-spawner.js`

Locate the wave scheduling loop. The spawner iterates over `waves` and uses `setTimeout` to schedule each enemy. After the last wave's last spawn resolves, fire one more timeout for the grace period:

```js
// After the final wave loop completes, calculate when the last enemy spawns:
const lastWave = waves[waves.length - 1];
const lastSpawnTime = lastWave.start + (lastWave.number - 1) * lastWave.interval;
const GRACE_MS = 2000;

setTimeout(() => {
  this.el.sceneEl.emit('waves-complete');
}, lastSpawnTime + GRACE_MS);
```

This fires once, unconditionally, after all wave scheduling is set up in `init()`.

### 2. Create `components/game-state.js`

```js
AFRAME.registerComponent('game-state', {
  init() {
    this.overlay = document.getElementById('game-over-overlay');
    this.overlay.style.display = 'none';

    this.el.sceneEl.addEventListener('waves-complete', () => {
      this.overlay.style.display = 'flex';
    });
  }
});
```

No schema needed for this issue. Issue 11 will extend `init()` with score listeners.

### 3. Add overlay and script — `index.html`

In `<head>`, after the other component scripts:
```html
<script src="components/game-state.js"></script>
```

In `<body>`, immediately before `<a-scene>`:
```html
<div id="game-over-overlay" style="
  display: none;
  position: fixed; inset: 0;
  background: rgba(0,0,0,0.85);
  color: #fff;
  font-family: monospace;
  flex-direction: column;
  align-items: center;
  justify-content: center;
  z-index: 999;
">
  <h1 style="font-size: 3em; margin-bottom: 0.5em;">GAME OVER</h1>
  <button
    onclick="location.reload()"
    style="font-size: 1.5em; padding: 0.5em 2em; cursor: pointer; background: #fff; color: #000; border: none; border-radius: 4px;">
    Play Again
  </button>
</div>
```

Note: `display: flex` is set by JS (not here as initial value) so it starts hidden regardless of CSS caching. When `waves-complete` fires, `style.display = 'flex'` activates the flexbox layout.

Attach `game-state` to the spawner entity:
```html
<a-entity enemy-spawner game-state></a-entity>
```

## Testing

| Scenario | Expected |
|----------|----------|
| Page loads | Overlay is hidden (`display: none`) |
| All waves complete + grace period passes | Overlay appears with "GAME OVER" heading and "Play Again" button |
| Click "Play Again" | Page reloads; overlay gone; game resets to initial state |
| Game over screen appearance on desktop | Centered overlay covers full viewport; readable font |
| Game over screen in VR headset | HTML overlay does **not** appear in headset (known limitation; acceptable for now) |
| `waves-complete` fires before all enemies are killed | Overlay still appears after grace period; remaining enemies continue until reload |

Manual test: open game in browser, wait for all waves to finish (or temporarily shorten wave timings in `enemy-spawner.js`), verify overlay appears after ~2 s.

## Risks / Edge Cases

- **VR visibility.** A fixed-position HTML `<div>` is not rendered inside the WebXR compositor. The game over screen will only be visible in desktop/2D mode. Full VR support would require a 3D A-Frame panel entity. Acceptable for this issue.

- **Grace period hardcoding.** The 2 s grace period is calculated relative to `lastWave.start + (lastWave.number - 1) * lastWave.interval`. If future issues add dynamic/infinite waves, this calculation will need to change. Keep it clearly commented.

- **`waves-complete` timing with remaining enemies.** The overlay fires based on spawn timing, not kill timing. Enemies spawned in the final wave may still be alive when the overlay appears. This is acceptable (screen shows when waves are done dispatching, not when all enemies are dead). A more accurate approach — waiting for all enemies to die — would require reference-counting live enemy entities and is out of scope here.

- **Interaction with issue 11 (point system).** Plan 11 also introduces `game-state.js` and the overlay. When implementing both, issue 10 provides the base `game-state` component and overlay HTML; issue 11 extends them with score display. Coordinate so issue 11 adds to `game-state.js` rather than replacing it.

- **`display: none` vs `display: flex` conflict.** The inline style sets `display: none` initially. JS sets it to `flex` on game over. Do not set `display: flex` in the inline style or the overlay will be visible on load.

- **Multiple `waves-complete` events.** The `setTimeout` that emits `waves-complete` fires once, but if `enemy-spawner` `init` is called multiple times (e.g., hot reload), the event could fire multiple times. Since the overlay just sets `display: flex`, duplicate events are harmless.
