# Plan: Issue 11 — Point System

## Summary

Add a point system where each enemy type awards a different number of points when killed. Track the running total in a central `game-state` component. Trigger game over when all waves complete (the enemy spawner already schedules everything via `setTimeout` chains, so a "done" event is straightforward to emit). Display the final score on a full-screen HTML overlay.

Point values:
- `fly-along-curve` (basic, 20 HP): **10 points**
- `shooting-enemy` (advanced, 30 HP, fires back): **25 points**

---

## Files to Change

| File | Change |
|------|--------|
| `components/enemy-spawner.js` | Stamp each spawned enemy entity with a `data-points` attribute; emit a custom `waves-complete` scene event after the last scheduled spawn |
| `components/destroyable.js` | On death, dispatch a `scene`-level `enemy-killed` custom event carrying `{ points }` read from the parent entity's `data-points` attribute |
| `components/game-state.js` | **New.** A-Frame component that accumulates score, listens for `waves-complete` to trigger game over, and shows/updates the overlay |
| `index.html` | Add `<script>` for `game-state.js`; attach `game-state` to a scene entity; add the game-over overlay `<div>` |

---

## Implementation Steps

### 1. Stamp point values on enemy entities — `enemy-spawner.js`

When creating each enemy entity, set `el.dataset.points` before appending:
- `fly-along-curve` spawns → `el.dataset.points = '10'`
- `shooting-enemy` spawns → `el.dataset.points = '25'`

After the very last `setTimeout` in the wave chain completes its final spawn, emit to the scene:
```js
this.el.sceneEl.emit('waves-complete');
```

### 2. Emit score event on death — `destroyable.js`

In the `die()` method, before removing the entity, read the point value from the parent and emit it up to the scene:
```js
const points = parseInt(this.el.parentEl.dataset.points || '0', 10);
if (points > 0) {
  this.el.sceneEl.emit('enemy-killed', { points });
}
```

The parent entity is the enemy root (the child `this.el` carries the `destroyable` component, the parent carries `data-points`).

### 3. Create `components/game-state.js`

Register an A-Frame component `game-state`:
- **Schema:** no user-configurable schema needed.
- **`init()`:**
  - Set `this.score = 0`
  - Cache the overlay DOM elements
  - Listen on `this.el.sceneEl` for `'enemy-killed'` → `onEnemyKilled(evt)`
  - Listen on `this.el.sceneEl` for `'waves-complete'` → `onWavesComplete()`
- **`onEnemyKilled(evt)`:** `this.score += evt.detail.points`
- **`onWavesComplete()`:** show the overlay and write the final score into it

### 4. Add overlay and script tag — `index.html`

Add a `<div id="game-over-overlay">` as a direct child of `<body>` (outside `<a-scene>`), styled to cover the viewport and hidden by default:

```html
<div id="game-over-overlay" style="
  display: none;
  position: fixed; inset: 0;
  background: rgba(0,0,0,0.85);
  color: #fff;
  font-family: monospace;
  display: flex; flex-direction: column;
  align-items: center; justify-content: center;
  z-index: 999;
">
  <h1>GAME OVER</h1>
  <p id="final-score">Score: 0</p>
  <button id="restart-btn" onclick="location.reload()">Play Again</button>
</div>
```

Use `display:none` initially (set via JS in `game-state.js` `init()` to ensure it starts hidden even if CSS is cached).

Add `<script src="components/game-state.js"></script>` to `<head>`.

Attach the component to any persistent entity — the enemy spawner entity is a natural fit:
```html
<a-entity enemy-spawner game-state></a-entity>
```

### 5. Waves-complete timing in `enemy-spawner.js`

The wave scheduler uses nested `setInterval`/`setTimeout`. After the last wave's last enemy is spawned, set a final timeout equal to a short grace period (e.g. 2 000 ms to let the last enemy be killed or animate away) before emitting `waves-complete`. This prevents the screen from flashing up while bullets are still in flight.

## Risks / Edge Cases

- **`destroyable` is on the model child, not the enemy root.** `this.el.parentEl` is used to read `data-points`. If the HTML structure ever nests the model deeper (e.g. a wrapper group), `parentEl` won't be the enemy root. Verify the actual entity hierarchy when testing. An alternative is to walk up with `closest('[data-points]')` if A-Frame exposes it, or traverse manually.

- **Interaction with plan 12 (powerup drops on death).** Plan 12 also modifies `destroyable.js`'s `die()` method. These changes must be merged carefully — both add logic at the same location. Order of operations doesn't matter here (emit event vs. spawn powerup entity), but the diff will conflict.

- **Interaction with plan 13 (damage multiplier) and plan 14 (laser).** Plan 14 adds a `takeDamage()` method to `destroyable.js`. This plan's change (`die()` emits an event) is additive and unlikely to conflict, but coordinate when both are implemented together.

- **`waves-complete` timing.** If future issues add infinite waves or dynamic wave generation, the hardcoded "last wave" detection will break. Keep the emission localized to the scheduler logic.

- **VR overlay visibility.** A fixed-position HTML `<div>` overlay is not rendered inside the WebXR compositor; it will not appear on the headset display. For full VR support, the score would need to be displayed as an A-Frame `<a-text>` or `<a-plane>` entity in 3D space. This plan targets the desktop/2D view as specified by the issue ("game over screen"); flag for future VR-specific work.

- **No lives/player-death trigger.** The issue only asks for points displayed at game over; game over is defined here as "all waves complete." If a separate issue later adds player lives, `game-state` should also listen for a `player-died` event and trigger the overlay then too.
