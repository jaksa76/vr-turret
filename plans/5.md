# Plan: Issue 5 — Two Turrets

## Summary

Allow the player to control the left and right turret independently using the left and right VR controllers. A toggle (A button on the right controller) switches between **dual mode** (each controller drives its own turret) and **mouse mode** (both turrets follow the mouse cursor as today). Mouse-only users are unaffected.

---

## Files to Change

| File | Change |
|---|---|
| `index.html` | Add `raycaster` to `#left-hand`; update `control-switcher` attributes |
| `components/control-switcher.js` | Rewrite to manage a dual/mouse mode toggle |
| `components/turret.js` | Remove conflicting hardcoded `abuttondown` listener |

## Implementation Steps

### 1. Add a raycaster to `#left-hand` (`index.html`)

`#left-hand` currently has no raycaster. The `turret` component reads `target.components.raycaster.intersections`, so the left hand needs one.

```html
<!-- before -->
<a-entity id="left-hand" laser-controls="hand: left" oculus-touch-controls="hand: left"></a-entity>

<!-- after -->
<a-entity id="left-hand" laser-controls="hand: left" oculus-touch-controls="hand: left"
    raycaster="objects: .colliding"></a-entity>
```

### 2. Update `control-switcher` placement and schema (`index.html`)

Move `control-switcher` off `#right-hand` (where it was a single-purpose button listener) and put it on a neutral entity, or keep it on `#right-hand` but extend its schema to know the full mapping. The simplest approach is to keep it on `#right-hand` and add schema properties that identify the two turrets and the two controllers:

```html
<a-entity id="right-hand"
    laser-controls="hand: right"
    oculus-touch-controls="hand: right"
    raycaster="objects: .colliding"
    control-switcher="leftTurret: #left-cannon; rightTurret: #right-cannon;
                      leftController: #left-hand; rightController: #right-hand">
</a-entity>
```

`#cursor` (mouse) remains the shared fallback and does not need to be in the schema (it is the well-known default).

### 3. Rewrite `control-switcher.js`

Replace the current single-event listener with a stateful component:

```js
AFRAME.registerComponent('control-switcher', {
    schema: {
        leftTurret:       { type: 'selector' },
        rightTurret:      { type: 'selector' },
        leftController:   { type: 'selector' },
        rightController:  { type: 'selector' }
    },
    init: function () {
        this.mode = 'mouse';   // 'mouse' | 'dual'

        this.el.addEventListener('abuttondown', () => {
            this.mode = (this.mode === 'mouse') ? 'dual' : 'mouse';
            this._applyMode();
        });
    },
    _applyMode: function () {
        const { leftTurret, rightTurret, leftController, rightController } = this.data;
        if (this.mode === 'dual') {
            leftTurret  && leftTurret.setAttribute('turret',  'target', '#' + leftController.id);
            rightTurret && rightTurret.setAttribute('turret', 'target', '#' + rightController.id);
        } else {
            leftTurret  && leftTurret.setAttribute('turret',  'target', '#cursor');
            rightTurret && rightTurret.setAttribute('turret', 'target', '#cursor');
        }
    }
});
```

Key points:
- Pressing A toggles between `mouse` and `dual`.
- In `dual` mode: `#left-cannon` → `#left-hand`, `#right-cannon` → `#right-hand`.
- In `mouse` mode: both cannons reset to `#cursor`.

### 4. Remove conflicting `abuttondown` listener from `turret.js`

`turret.js` currently has its own `abuttondown` handler that unconditionally switches the target to `#right-hand` for whichever cannon it's attached to. This will fight with the new toggle logic.

Remove lines 8–10:

```js
// DELETE these lines from turret.js init:
this.el.addEventListener('abuttondown', () => {
    this.el.setAttribute('turret', 'target', '#right-hand');
});
```

The `turret` component's job is purely to aim; mode management belongs in `control-switcher`.

## Testing

There is no automated test suite. Manual testing scenarios:

| Scenario | Expected behaviour |
|---|---|
| Load in browser (desktop/mouse) | Both turrets track mouse cursor. No change from today. |
| Load in VR, do not press A | Both turrets track mouse cursor (default `mouse` mode). |
| Press A in VR | Switch to `dual` mode. Left turret follows left controller laser; right turret follows right controller laser. Each aims and fires independently. |
| Press A again in VR | Switch back to `mouse` mode. Both turrets follow mouse cursor again. |
| Press A repeatedly | Mode alternates correctly each press without stale targeting. |
| Fire with dual mode | Each turret fires projectiles toward its respective controller aim point. |
| Enemy hit detection | `destroyable` / physics collision still works for projectiles from both turrets in both modes. |

## Risks / Edge Cases

1. **`raycaster.intersections` empty on left hand** — If the left controller laser does not intersect `.colliding` geometry, the turret component will not rotate. This is pre-existing behaviour shared with the right hand; the left hand's laser just needs pointing at a target. No code change needed, but worth noting in UX.

2. **`abuttondown` fires on cannon entity vs controller entity** — The old `turret.js` listener fires on the cannon (child of turret group). In VR, `oculus-touch-controls` events bubble, but the cannon entity may not receive them reliably. The new `control-switcher` listener on the right-hand controller entity is the authoritative source; the old cannon listener in `turret.js` must be removed to avoid conflicts.

3. **Selector vs ID string in `control-switcher`** — A-Frame `type: 'selector'` resolves the DOM element at init. If entities are added dynamically later, re-calling `update` may be needed. In this scene all entities are static, so this is safe.

4. **Both controllers pointing at the same target** — Not a problem; turrets track independently, and projectiles from both will collide with enemies correctly.

5. **Mode persistence across scene reload** — Mode resets to `mouse` on every page load, which is the sensible default. No persistence needed.

6. **`laser-controls` visual ray direction** — The `laser-controls` component draws a visible ray from the controller. In dual mode both lasers show independently, which is the desired UX. No extra work needed.
