# Plan: Issue 7 — Kamikaze Enemies

## Summary

Add a new `kamikaze-enemy` A-Frame component that has two distinct movement phases:

1. **Approach phase** — follows a `CatmullRomCurve3` path slowly toward the player area.
2. **Rush phase** — once the enemy comes within a configurable `threshold` distance of the player/camera, it abandons the curve and flies directly at the player at high speed.

On contact (within a small `explodeRadius` of the player), the enemy explodes and — if the energy shield from plan 9 is present — deals configurable `damage` to the shield. If no shield component exists, the explosion is purely visual. The enemy can also be shot down by bullets during either phase (via the existing `destroyable` component on the child model entity).

Configurable per-wave parameters: `speed` (approach), `rushSpeed`, `threshold` (rush trigger distance), `explodeRadius`, `damage`, and `model` (gltf model asset id or src URL for the ship mesh).

## Implementation Steps

### 1. Create `components/kamikaze-enemy.js`

**Schema:**

```js
schema: {
  curve:        { type: 'string',  default: '0 -30 -80, 0 0 -50, 0 15 0' },
  speed:        { type: 'number',  default: 0.00005  }, // curve progress units/ms (approach)
  rushSpeed:    { type: 'number',  default: 0.015    }, // world-units/ms (rush)
  threshold:    { type: 'number',  default: 20       }, // world-units; distance to trigger rush
  explodeRadius:{ type: 'number',  default: 3        }, // world-units; contact detonation range
  damage:       { type: 'number',  default: 50       }, // HP damage dealt to player shield on contact
}
```

**`init()`**
- Parse `data.curve` into a `THREE.CatmullRomCurve3` (same split/map pattern as `fly-along-curve.js`).
- Set `this.state = 'approach'`, `this.spawnTime = null`, `this.rushDir = new THREE.Vector3()`.
- Cache the player target: `this.target = document.querySelector('#camera-rig') ?? document.querySelector('[camera]')`.
- Bind a `_onDead` listener on `this.el` for the `enemy-dead` event (see step on child death forwarding).

**`tick(time, timeDelta)`**

```
// Lazy-init spawn time
if (!this.spawnTime) this.spawnTime = time;

if (this.state === 'approach') {
  d = (this.data.speed * (time - this.spawnTime)) % 0.98;
  pos = curve.getPointAt(d);           // THREE.Vector3
  next = curve.getPointAt((d + 0.01) % 0.98);
  this.el.object3D.position.copy(pos);
  // face forward along curve (same atan2/asin as fly-along-curve)
  ...

  // Check rush threshold
  targetPos = new THREE.Vector3();
  this.target.object3D.getWorldPosition(targetPos);
  if (pos.distanceTo(targetPos) < this.data.threshold) {
    this.state = 'rush';
    this.rushDir.copy(targetPos).sub(pos).normalize();
  }

} else if (this.state === 'rush') {
  // Move toward player
  this.el.object3D.position.addScaledVector(this.rushDir, this.data.rushSpeed * timeDelta);

  // Re-aim at player each tick so the rush tracks movement
  targetPos = new THREE.Vector3();
  this.target.object3D.getWorldPosition(targetPos);
  this.rushDir.copy(targetPos).sub(this.el.object3D.position).normalize();

  // Face direction of movement
  const pos = this.el.object3D.position;
  this.el.object3D.lookAt(pos.clone().add(this.rushDir));

  // Contact check
  if (pos.distanceTo(targetPos) < this.data.explodeRadius) {
    this._detonate();
  }
}
```

**`_detonate()`**
- Spawn an explosion entity at the current world position (same pattern as `destroyable.js`):
  ```js
  const boom = document.createElement('a-entity');
  boom.setAttribute('explosion', '');
  boom.setAttribute('position', positionString);
  this.el.sceneEl.appendChild(boom);
  ```
- If a shield component is present on the camera/rig, emit a damage event or call its method:
  ```js
  const shield = document.querySelector('[energy-shield]');
  if (shield) shield.components['energy-shield'].takeDamage(this.data.damage);
  ```
- Remove the parent entity: `this.el.parentNode.removeChild(this.el)`.

**Child death forwarding:** The `destroyable` component lives on the child model entity and removes *that* entity on death. Add an event listener in `init()`:
```js
this.el.addEventListener('child-destroyed', () => this._detonate());
```
Actually, the simpler approach is to give the model child a custom `destroyable` that also emits an event on the parent. Or override: after `destroyable` fires its own removal, emit `kamikaze-dead` on the parent. The cleanest solution is to listen for the child's `componentremoved` / entity removal. Since `destroyable` calls `this.el.parentNode.removeChild(this.el)` (child), the parent entity still exists. Add a `MutationObserver` on the parent's children, or simpler: check each tick whether the child is still in the DOM:
```js
// In tick, after movement:
if (!this.modelEl.parentNode) { this._detonate(); return; }
```
Where `this.modelEl` is the child model entity cached in `init()` via `this.el.querySelector('[destroyable]')` after the first tick.

## Testing

Testing is manual (no automated test suite):

1. **Approach phase** — Enemy should follow the curve path smoothly at `speed`. Rotation should face the curve's forward direction, matching `fly-along-curve` behaviour.
2. **Rush trigger** — When the enemy enters the `threshold` radius, it should visibly accelerate and turn directly toward the player/camera. Confirm the transition is immediate (no lag frame).
3. **Rush tracking** — While rushing, the enemy should re-aim each tick; if the player moves (in VR), the enemy should curve toward the new position.
4. **Contact explosion** — Allow a rushing enemy to reach the player. Verify the explosion visual fires at the correct world position and the enemy entity is removed.
5. **Bullet kill during approach** — Shoot the enemy while it is still on the curve. Confirm `destroyable` fires, the child is removed, `_detonate()` triggers (explosion spawned), and no orphan parent entity lingers.
6. **Bullet kill during rush** — Shoot the enemy mid-rush. Same as above.
7. **Config extremes:**
   - `threshold: 0` — enemy should never enter rush until it is almost on top of the player.
   - `threshold: 200` — enemy should rush immediately on spawn.
   - `rushSpeed: 0.001` — very slow rush (visually crawls).
   - `rushSpeed: 0.1` — near-instant arrival.
8. **Custom model** — Set `wave.model` to a different asset id (e.g. `#enemy-ship`) and verify the mesh switches.
9. **Multiple simultaneous kamikazes** — Spawn 5 at once; confirm independent state machines, no cross-entity corruption.
10. **Shield integration (if plan 9 is implemented)** — Confirm `damage` value reduces shield HP on contact; confirm `0` damage config causes no crash.

## Risks / Edge Cases

- **Target not found at init:** If `#camera-rig` and `[camera]` are both absent at `init()` time (rare during hot-reload), `this.target` is null. Guard `tick()` with an early return and retry the lookup each tick until found.
- **Child model not yet attached at first tick:** `this.el.querySelector('[destroyable]')` called in `init()` may return `null` because the child entity is appended *after* `kamikaze-enemy` initialises. Cache `modelEl` lazily inside `tick()` on first use.
- **`_detonate()` called twice:** If the child dies from bullets *and* the contact distance check fires in the same tick, `_detonate()` could run twice. Guard with a `this.dead` boolean flag set on first call.
- **World-space distance:** The enemy parent entity is a direct child of the scene; `this.el.object3D.position` is already in world space. The camera rig may be nested. Always use `getWorldPosition()` for the target.
- **lookAt flips at 180°:** `Object3D.lookAt` can produce a sudden flip when the rush direction passes directly overhead. This is cosmetic but noticeable. Clamping or using `slerp` toward the target orientation each tick smooths it.
- **Curve loop boundary:** `% 0.98` used in `fly-along-curve` to avoid the seam. Apply the same cap in the approach phase to prevent a position jump at loop wrap.
- **No player hitbox:** Detection is distance-based, not physics-based, because the player has no `ammo-body`. This is intentional but means the explosion range feels approximate. `explodeRadius: 3` should feel fair at normal scale.
- **`explosion` component dependency:** `_detonate()` spawns `<a-entity explosion>`. If `explosion.js` is not loaded or the component name differs, the spawn silently does nothing. Confirm the component name matches `components/explosion.js`.
- **Performance during rush:** Re-computing `getWorldPosition` and `distanceTo` every tick for multiple rushing kamikazes is cheap, but avoid adding querySelectorAll calls in the rush path.
