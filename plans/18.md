# Plan: Issue 18 — Enemy Wave Generation System

## Summary

Replace the hardcoded `waves` array in `enemy-spawner.js` with an algorithmic generator that produces a full level of enemy waves from a small set of parameters. Generation is deterministic via a seeded PRNG (mulberry32) so the same seed always reproduces the same level. Difficulty, wave count, and timing are all driven by parameters.

The implementation splits into two pieces:

1. **`components/wave-generator.js`** — pure JS `WaveGenerator` class (no A-Frame dependency) containing the PRNG and generation logic, plus a thin A-Frame component wrapper that exposes it to the scene.
2. **`enemy-spawner.js` (modified)** — checks at `init` whether a `wave-generator` component is present on the same entity and, if so, reads the generated waves instead of the hardcoded constant.

`index.html` gets a new `<script>` tag and the spawner entity gains `wave-generator` attributes. The hardcoded `waves` constant in `enemy-spawner.js` is kept as a fallback so nothing breaks if `wave-generator` is absent.

## Implementation Steps

### 1. Create `components/wave-generator.js`

#### 1a. Mulberry32 PRNG

A simple, fast, seeded 32-bit PRNG. Takes a seed integer, returns a function that produces floats in `[0, 1)` each call.

```js
function mulberry32(seed) {
    return function() {
        seed |= 0; seed = seed + 0x6D2B79F5 | 0;
        let t = Math.imul(seed ^ seed >>> 15, 1 | seed);
        t = t + Math.imul(t ^ t >>> 7, 61 | t) ^ t;
        return ((t ^ t >>> 14) >>> 0) / 4294967296;
    };
}
```

#### 1b. `WaveGenerator` class

Constructor accepts a params object:

| Param | Default | Description |
|---|---|---|
| `seed` | `1` | Integer seed for PRNG |
| `numWaves` | `10` | Total waves to generate |
| `difficulty` | `0.5` | 0–1 scale; drives enemy count, health, speed, fire rate |
| `waveGap` | `6000` | Base ms gap between wave starts |
| `sceneDepth` | `80` | Max Z distance for curve endpoints (positive = depth away from player) |
| `sceneWidth` | `16` | ±X range for curve control points |
| `sceneHeight` | `10` | ±Y range for curve control points |

`generate()` method returns an array of wave definition objects compatible with the existing `enemy-spawner.js` schema.

**Per-wave logic:**

- **`start`**: `waveGap * i` plus a small PRNG jitter (±10 % of waveGap)
- **`number`**: `Math.max(1, Math.round(2 + i * 0.5 * difficulty * 3 + rand() * 2))`  — grows with wave index and difficulty
- **`interval`**: `800 - difficulty * 400 + rand() * 200` ms — tighter spawns at higher difficulty
- **`type`**: first two waves are always `fly-along-curve`; from wave 3 onward a random draw (weighted by difficulty) may yield `shooting-enemy`
- **`curve`**: 4–6 PRNG-generated control points (see §1c)
- For `shooting-enemy` waves: `stopProgress` in `[0.35, 0.65]`, `fireRate` `1 + difficulty * 3`, `health` `1 + Math.floor(difficulty * 4)`

#### 1c. Curve generation helper

`generateCurve(rand, params)`:
- Always starts with a distant point: `(rand*sceneWidth - sceneWidth/2, rand*sceneHeight/2, -sceneDepth)`
- Generates 2–4 mid-points distributed across the scene volume
- Ends with another distant point on the opposite side so enemies sweep across
- Returns the curve as a comma-separated `"x y z"` string matching the existing format

Example:
```
-6 2 -80, -3 -1 -50, 4 3 -30, 7 -2 -80
```

#### 1d. A-Frame component wrapper

```js
AFRAME.registerComponent('wave-generator', {
    schema: {
        seed:        { type: 'int',    default: 1 },
        numWaves:    { type: 'int',    default: 10 },
        difficulty:  { type: 'number', default: 0.5 },
        waveGap:     { type: 'number', default: 6000 },
        sceneDepth:  { type: 'number', default: 80 },
        sceneWidth:  { type: 'number', default: 16 },
        sceneHeight: { type: 'number', default: 10 }
    },
    init() {
        const gen = new WaveGenerator(this.data);
        this.waves = gen.generate();
    }
});
```

`enemy-spawner` will read `this.waves` from this component.

### 2. Modify `enemy-spawner.js`

In `init()`, before the existing `for (let wave of waves)` loop, add:

```js
const generatorComponent = this.el.components['wave-generator'];
const waveDefs = generatorComponent ? generatorComponent.waves : waves;
```

Replace all references to `waves` in the spawning loop with `waveDefs`.

The hardcoded `waves` constant at the top of the file is kept unchanged as a fallback for manual scene configuration and as a reference example.

---

### 3. Update `index.html`

Add the script tag (before `enemy-spawner.js`):

```html
<script src="components/wave-generator.js"></script>
```

Update the spawner entity:

```html
<!-- before -->
<a-entity enemy-spawner></a-entity>

<!-- after -->
<a-entity
    wave-generator="seed: 42; numWaves: 10; difficulty: 0.4; waveGap: 6000"
    enemy-spawner>
</a-entity>
```

`wave-generator` component runs its `init` first (A-Frame processes components in declaration order), so `enemy-spawner.init` can safely read `this.el.components['wave-generator'].waves`.

## Testing

No automated test suite exists in this repo. Manual testing scenarios:

| Scenario | Expected |
|---|---|
| Load page with `wave-generator` attributes | Console shows no errors; enemies spawn over time according to generated waves |
| Change `seed` value | Different arrangement of enemies, different curves, different types |
| Same `seed` on two reloads | Identical enemy sequence both times (determinism) |
| `difficulty: 0.1` | Small waves, long gaps, no or few shooting enemies, low health |
| `difficulty: 0.9` | Large waves, short gaps, many shooting enemies, higher health |
| `numWaves: 3` | Exactly 3 waves spawn then nothing |
| Remove `wave-generator` from entity | Falls back to hardcoded `waves` constant; original two waves spawn as before |
| Shooting enemies in generated waves | Enemy flies curve to `stopProgress`, stops, aims at camera, fires |
| Fly-along enemies in generated waves | Enemy follows generated curve, loops if it reaches the end |
| Generated curves with debug mode | `fly-along-curve="debug: true"` draws red curve lines in expected locations |

To assist manual testing, `WaveGenerator` can be exercised in the browser console:

```js
const gen = new WaveGenerator({ seed: 42, numWaves: 3, difficulty: 0.5 });
console.log(JSON.stringify(gen.generate(), null, 2));
```

## Risks / Edge Cases

1. **A-Frame component init order** — A-Frame processes components in the order they appear in the attribute list. `wave-generator` must be listed before `enemy-spawner` in the HTML to guarantee its `init` runs first. If the order is ever swapped, `enemy-spawner` will silently fall back to the hardcoded waves (acceptable, but worth a comment in the code).

2. **Degenerate curves** — If PRNG-generated control points happen to be nearly collinear or very close together, `CatmullRomCurve3` may produce strange paths (tight loops, near-zero length segments). Mitigate by enforcing a minimum XZ spread between consecutive points during generation.

3. **Enemies spawning inside geometry** — Generated control points might intersect scene geometry. The existing scene is mostly open space, so this is low risk, but worth noting if scene layout changes.

4. **Very high `numWaves` / `difficulty`** — At extreme values, later waves may spawn hundreds of enemies, tanking framerate. A sanity cap (e.g. `maxEnemiesPerWave: 20`) should be enforced inside `generate()`.

5. **`shooting-enemy` `target` selector** — The generated wave definitions should default `target` to `'#camera'` (matching the existing hardcoded wave) unless overridden. The generator must always emit this field for shooting-enemy waves.

6. **PRNG seed as integer** — A-Frame's `type: 'int'` truncates floats. Passing a float seed via HTML attributes will silently round it. Document that `seed` must be an integer.

7. **No wave-completion gating** — This system uses time-based scheduling (same as the existing spawner) and does not wait for all enemies from one wave to die before starting the next. This is consistent with the current design; a kill-gated system would be a larger follow-on change.
