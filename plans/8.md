# Plan: Issue 8 â€” Dodging Enemies

## Summary

Add a new `dodging-enemy` A-Frame component that follows a curve path like `fly-along-curve` but actively detects incoming player bullets and sidesteps them. The dodge speed and reaction time are configurable schema properties. The enemy is registered as a new wave type in `enemy-spawner.js`.

**Dodge detection approach:** Each tick (throttled by `reactionTime`), scan all `.bullet` elements in the scene. For each bullet, compute whether it is heading toward the enemy (positive dot product of bullet velocity with the direction from bullet to enemy, within a detection radius). When a threat is detected, apply a lateral offset perpendicular to the curve's forward direction. The offset decays back to zero over time so the enemy returns to its base path after evading.

## Implementation Steps

### 1. Create `components/dodging-enemy.js`

Define the component schema:

```js
schema: {
  curve:          { /* same parse/stringify as fly-along-curve */ },
  speed:          { type: 'number', default: 0.0001 },
  dodgeSpeed:     { type: 'number', default: 5 },      // world-units/sec lateral speed
  reactionTime:   { type: 'number', default: 300 },    // ms delay before dodge begins
  detectionRadius:{ type: 'number', default: 15 },     // world-units; bullet must be this close
  debug:          { type: 'boolean', default: false },
}
```

**`init()`**
- Build `THREE.CatmullRomCurve3` from `data.curve` (same as `fly-along-curve`).
- Initialise state: `spawnTime = null`, `dodgeOffset = new THREE.Vector3()`, `pendingDodge = null`, `lastScanTime = 0`.

**`tick(time, timeDelta)`**

_Curve following (identical to `fly-along-curve`):_
```
d = (speed * (time - spawnTime)) % 0.98
basePos = curve.getPointAt(d)
forward = (curve.getPointAt(d + 0.01) - basePos).normalize()
```

_Bullet scan (throttled to once per `reactionTime` ms):_
```
if (time - lastScanTime >= reactionTime):
  lastScanTime = time
  for each .bullet element:
    bulletPos = bullet.object3D.position (world)
    toEnemy = basePos - bulletPos
    dist = toEnemy.length()
    if dist < detectionRadius:
      bulletVel = bullet components['projectile'].data.velocity  (or read from component)
      if toEnemy.dot(bulletVel) > 0:   // bullet heading toward enemy
        schedule a dodge in reactionTime ms (if not already pending)
```

_Dodge execution:_
- When the reaction timer fires, pick a lateral direction: cross product of `forward` and world-up `(0,1,0)`, normalised. Randomly choose left or right (or always pick the direction away from the bullet).
- Set `dodgeTarget = Â±lateral * dodgeSpeed * someFixedDuration` â€” a target offset.
- Each tick, lerp `dodgeOffset` toward `dodgeTarget` at `dodgeSpeed * timeDelta`; once reached, lerp back to zero.

_Apply final position:_
```
entity.object3D.position = basePos + dodgeOffset
```

_Rotation:_ same atan2/asin formula as `fly-along-curve`.

**Bullet velocity access:** `projectile.js` stores velocity as `this.data.velocity` on the `projectile` component. Access via `bulletEl.components['projectile'].data.velocity` (a `THREE.Vector3`-compatible object `{x,y,z}`).

## Testing

Since there is no automated test suite, testing is manual in-browser / in-headset:

1. **Basic movement** â€” Enemy should follow the curve exactly like a `fly-along-curve` enemy when no bullets are nearby.
2. **Dodge triggers** â€” Fire the turret at a dodging enemy; it should sidestep before the bullet arrives (visible lateral movement). Verify the delay matches `reactionTime`.
3. **Returns to path** â€” After dodging, the enemy should smoothly return to the base curve position.
4. **Config range** â€” Test extreme values:
   - `reactionTime: 0` â€” immediate dodge (may feel jittery).
   - `reactionTime: 2000` â€” enemy barely reacts.
   - `dodgeSpeed: 0` â€” no dodge (enemy stays on path).
   - `dodgeSpeed: 30` â€” very fast dodge.
5. **Multiple bullets** â€” Rapid-fire multiple bullets; enemy should only schedule one pending dodge at a time (no re-triggering until previous dodge resolves).
6. **Detection radius** â€” With `detectionRadius: 1`, enemy should never dodge. With `detectionRadius: 50`, it should dodge bullets that pose no real threat.
7. **No false positives** â€” Enemy bullets from `shooting-enemy` should not trigger the dodge (`.bullet` selector is player-only; confirm class name exclusivity).

## Risks / Edge Cases

- **Bullet velocity access:** `projectile.js` stores velocity as raw `{x,y,z}` in its schema. If the component hasn't initialised yet when the dodging enemy tries to read it, `bulletEl.components['projectile']` could be undefined. Guard with a null-check.
- **World vs. local coordinates:** `fly-along-curve` positions entities directly in scene world space. Ensure bullet positions are also sampled in world space (`getWorldPosition`) since they may be children of other entities.
- **Dodge offset accumulation:** If the enemy receives bullets continuously, the offset must be clamped to a maximum magnitude so the enemy doesn't teleport far off-screen.
- **Curve end wrap-around:** The curve loops via `% 0.98`. Ensure lateral dodge doesn't produce a nonsensical forward vector near the seam (d â‰ˆ 0.97â€“0.98).
- **Performance:** Querying `document.querySelectorAll('.bullet')` every tick can be expensive when many bullets exist. The `reactionTime` throttle mitigates this; keep `reactionTime â‰¥ 100 ms`.
- **Collision box mismatch:** The ammo-shape is centred on the model child entity, not the parent. The lateral dodge moves the parent; the physics body follows. No change needed, but note that physics collision detection still fires correctly because the kinematic body is on the child and ammo tracks its world transform.
- **Enemy bullet class conflict:** Verify that `shooting-enemy` projectiles do NOT have `class="bullet"`. If they do, dodging enemies will try to evade them (may be desirable, but should be intentional).
