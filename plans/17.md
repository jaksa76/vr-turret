# Plan: Issue 17 â€” Secondary Weapon: Bombs

## Summary

Add a bomb secondary weapon that fires from the turret alongside the primary laser. Bombs travel in a straight line but explode via **proximity detection** (not physics collision), applying configurable area-of-effect damage to all `destroyable` enemies within the blast radius. AoE radius, damage per hit, and proximity trigger distance are all configurable on the component.

The implementation requires three changes:
1. A new `bomb.js` component â€” handles movement and proximity/AoE logic.
2. A new `bomb-launcher.js` component â€” fires bombs at a slower auto-fire rate (secondary cadence).
3. A small update to `destroyable.js` â€” adds a `bomb-damage` event listener so enemies can be damaged without physics collision.

## Implementation Steps

### Step 1 â€” Extend `destroyable.js` to handle bomb damage

Add a `bomb-damage` event listener alongside the existing `collidestart` listener:

```js
init: function() {
    this.el.addEventListener('collidestart', this.hit.bind(this));
    this.el.addEventListener('bomb-damage', this.onBombDamage.bind(this));
},
onBombDamage: function(evt) {
    this.data.health -= evt.detail.damage;
    if (this.data.health <= 0) this.die();
},
```

This keeps bomb damage decoupled from physics and avoids touching the existing bullet path.

### Step 2 â€” Create `components/bomb.js`

Schema:
```js
schema: {
    velocity:        { type: 'vec3',   default: {x:0, y:0, z:-0.03} },
    proximityRadius: { type: 'number', default: 5  },
    aoeRadius:       { type: 'number', default: 8  },
    damage:          { type: 'number', default: 50 },
}
```

`tick` logic:
1. Advance position by `velocity * timeDelta` (same as `projectile.js`).
2. Remove bomb if `position.z < -500`.
3. Query all `[destroyable]` elements in the scene (attribute selector â€” enemies use the `destroyable` component, not a CSS class).
4. For each, compute world-space distance to the bomb.
5. If any distance â‰¤ `proximityRadius` â†’ call `this.explode()`.

`explode()`:
1. Guard against double-detonation (`this.exploded` flag).
2. Query all `[destroyable]` elements again; emit `bomb-damage` on those within `aoeRadius`.
3. Spawn an `explosion` entity at the bomb's world position (reuse existing component; use a larger `radius` than the default bullet explosion).
4. Remove the bomb entity from the scene.

No physics body needed on the bomb â€” proximity is checked manually, which avoids ammo.js overlap / tunneling issues at high speed.

### Step 3 â€” Create `components/bomb-launcher.js`

Schema:
```js
schema: {
    fireRate:        { type: 'number', default: 3000 },  // ms between bombs
    speed:           { type: 'number', default: 0.03  },
    proximityRadius: { type: 'number', default: 5     },
    aoeRadius:       { type: 'number', default: 8     },
    damage:          { type: 'number', default: 50    },
}
```

`init`: set `this.lastShot = 0`.

`tick` logic (mirrors `shooter.js`):
1. If `time - this.lastShot < this.data.fireRate` â†’ return.
2. Sample world position and quaternion from `this.el.object3D`.
3. Compute velocity vector `(0, 0, -speed)` rotated by quaternion.
4. Create `a-entity`, set `bomb` component with all schema values forwarded, set `position`, set a visible geometry (sphere, radius ~0.15), add class `bomb` for easy querying.
5. Append to scene. Update `this.lastShot`.
6. Optionally play a distinct sound (can reuse `explosionSound` at low volume or add a new asset).

### Step 4 â€” Wire up in `index.html`

Add script tags (before closing `</head>`):
```html
<script src="components/bomb.js"></script>
<script src="components/bomb-launcher.js"></script>
```

Add `bomb-launcher` to the shooter entity inside each cannon (or as a sibling entity at the same barrel position). Example:
```html
<a-entity shooter="" bomb-launcher="" sound="src: #laserShoot; poolSize: 2; volume: 0.2" position="0 0.33 -1.21"></a-entity>
```

Enemies use the `destroyable` **component attribute** (not a CSS class), so the bomb's query must use `document.querySelectorAll('[destroyable]')` â€” no changes to enemy-spawner needed.

## Risks / Edge Cases

- **Performance of per-tick querySelectorAll**: Querying `.destroyable` every tick for every live bomb could be expensive if many bombs or enemies are present. Mitigate by caching the node list and refreshing it only when the DOM changes, or by limiting query frequency (e.g., every 100 ms).
- **Double die() on destroyable**: If multiple bombs explode simultaneously near the same enemy, `bomb-damage` may fire more than once before `this.data.dead` is set. The existing `die()` guard (`if (this.data.dead) return`) already covers this â€” ensure `onBombDamage` also exits early when `this.data.dead` is true.
- **No physics body on bomb**: Bombs won't collide with terrain or the player's ship model. If that's desired later, a kinematic body can be added, but it's out of scope for this issue.
- **Explosion visual radius**: The existing `explosion` component's `radius` is visually scaled. Pass a larger radius (e.g., `aoeRadius / 2`) to make the blast look proportional.
- **Enemy selector**: Enemies have `destroyable` as a component attribute, not a CSS class. The query must be `querySelectorAll('[destroyable]')`. Using `.destroyable` would silently return no results and bombs would never explode.
- **Bomb visible geometry**: A sphere is suggested; ensure it doesn't visually look like a laser bullet. A darker color (e.g., `#444`) with a slightly larger radius (~0.15) distinguishes it well.
