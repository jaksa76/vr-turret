# Plan: Better Sprite for Projectile

## Summary

Replace the cube geometry used for player projectiles with billboard sprites (textured planes that always face the camera). The sprite image will be configurable on the `shooter` component and must support alpha-channel transparency for PNG/GIF images.

## Files to Change

- `components/shooter.js` — add `sprite` and `spriteSize` schema properties; when `sprite` is set, create a plane with a flat, transparent material instead of a box.
- `components/projectile.js` — add billboard behavior: cache a reference to the camera in `init`, then `lookAt` the camera each tick so the sprite always faces the viewer.
- `index.html` — declare a default bullet sprite in `<a-assets>` and pass `sprite` to both `<a-entity shooter>` elements.

A new sprite image (e.g. `images/bullet.png`, a simple round laser bolt with alpha) should be added to the repo; alternatively the existing `images/particle.gif` can be used as a placeholder during development.

## Implementation Steps

1. **`shooter.js` schema** — add two properties:
   ```js
   sprite: { type: 'string', default: '' },   // asset selector or URL; empty = legacy cube
   spriteSize: { type: 'number', default: 0.15 }
   ```

2. **`shooter.js` projectile creation** — branch on `this.data.sprite`:
   - If empty: keep existing `geometry: box` + `ammo-shape: box` (backwards-compatible).
   - If set: use `geometry: {primitive: 'plane', width: spriteSize, height: spriteSize}` and set material with `src`, `transparent: true`, `alphaTest: 0.5`, `shader: flat`, `side: double`. Keep `ammo-body: kinematic` and `ammo-shape: box` (physics collider can remain a box even if visual is a sprite).

3. **`projectile.js` init** — cache camera: `this.camera = document.querySelector('[camera]');`

4. **`projectile.js` tick** — after moving the entity, if the camera reference exists, call:
   ```js
   this.el.object3D.lookAt(this.camera.object3D.getWorldPosition(new THREE.Vector3()));
   ```
   This makes the plane always face the camera (billboard effect).

5. **`index.html` asset** — add an `<img>` asset tag for the bullet sprite:
   ```html
   <img id="bullet-sprite" src="images/bullet.png">
   ```
   (Add a suitable `bullet.png` with alpha to `images/`.)

6. **`index.html` shooter attributes** — add `sprite: #bullet-sprite` to both `<a-entity shooter>` elements.

## Testing

- **Visual check (desktop):** Open `index.html` in a browser; fired projectiles should appear as flat sprites oriented toward the camera, not as cubes.
- **Billboard rotation:** Move the camera left/right; the sprites should always face the camera (no edge-on slivers visible).
- **Alpha channel:** Use an image with a transparent background; the background pixels should be invisible (verify no black square around the sprite).
- **Collision still works:** Projectiles must still register `collidestart` with enemies — the physics collider is unchanged, so this should continue to work.
- **Backwards compatibility:** A `shooter` element without the `sprite` attribute should still fire cube projectiles (default empty string takes the old code path).
- **VR headset:** Confirm sprites are visible and billboard-facing on the Meta Quest.

## Risks / Edge Cases

- **`lookAt` and physics:** Calling `lookAt` on an `ammo-body: kinematic` entity overrides any physics-driven rotation. Since kinematic bodies are user-controlled this is fine, but if the physics system resets the quaternion after our `lookAt` we may need to move the billboard call to after the physics step or use a separate child entity for the visual.
- **Camera reference timing:** `document.querySelector('[camera]')` in `init` may resolve before the camera entity is fully initialized. Guard with a null check in tick and re-query if needed.
- **Performance of `new THREE.Vector3()` in tick:** Allocating a vector every tick per projectile creates GC pressure. Reuse a pre-allocated `_v3` scratch vector stored on `this`.
- **`alphaTest` value:** `0.5` is a common default but may clip semi-transparent edges. The threshold should be tuned to the specific sprite art; a lower value (e.g. `0.1`) preserves soft edges.
- **`side: double`:** Without double-sided rendering, if the billboard overshoots 90° the plane becomes invisible. Double-sided prevents this but at a slight render cost.
